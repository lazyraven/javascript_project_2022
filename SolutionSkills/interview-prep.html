JavaScript — Core Basics

Q1 What is the difference between var, let, and const?
Basic▾
Answer

var is function-scoped and hoisted (initialized as undefined). It can be re-declared and updated. let is block-scoped, hoisted but not initialized (Temporal Dead Zone), and can be updated but not re-declared. const is block-scoped and must be initialized at declaration — the binding cannot be reassigned (but object contents can still be mutated).

// var hoisting quirk
console.log(x);       // undefined (hoisted)
var x = 5;

// let — Temporal Dead Zone
// console.log(y);     // ReferenceError
let y = 10;

// const — object mutation still allowed
const obj = { a: 1 };
obj.a = 2;            // ✓ OK
// obj = {};          // ✗ TypeError
Interviewer tip: Prefer const by default, use let only when you need to reassign, and avoid var in modern code.
Q2 What is hoisting in JavaScript?
Basic▾
Answer

Hoisting is JavaScript's behavior of moving declarations to the top of their scope during the compilation phase. Function declarations are fully hoisted (callable before declaration). var declarations are hoisted and initialized to undefined. let/const are hoisted but not initialized — accessing them before their declaration throws a ReferenceError (Temporal Dead Zone).

greet();              // "Hello!" — function declaration hoisted
function greet() { console.log("Hello!"); }

console.log(name);    // undefined — var hoisted, not yet assigned
var name = "Alice";

// sayHi();            // TypeError — function expression NOT hoisted
var sayHi = function() { console.log("Hi"); };
Q3 Explain == vs ===. When would you use each?
Basic▾
Answer

== performs abstract equality — it coerces types before comparing. === performs strict equality — no type coercion, both value and type must match.

0 == ""        // true  (both coerced to 0)
0 === ""       // false (number ≠ string)
null == undefined  // true  (special case)
null === undefined // false
1 == true     // true  (true coerced to 1)
1 === true    // false
Best practice: Always use ===. The only common exception: x == null to check for both null and undefined.
Q4 What are JavaScript's data types? What's the difference between primitive and reference types?
Basic▾
Answer

Primitives (stored by value): string, number, boolean, null, undefined, symbol, bigint. Reference types (stored by reference in heap): object, array, function.

// Primitives — copied by value
let a = 5, b = a;
b = 10;
console.log(a); // 5 (unchanged)

// Objects — copied by reference
let x = { val: 5 };
let y = x;
y.val = 10;
console.log(x.val); // 10 (same reference)

// Deep clone to avoid mutation
let z = JSON.parse(JSON.stringify(x));
// or: structuredClone(x)  (modern)

JavaScript — Closures & Scope

Q5 What is a closure? Give a practical example.
Intermediate▾
Answer

A closure is a function that remembers its lexical environment even when executed outside of it. The inner function has access to the outer function's variables after the outer function has returned.

function makeCounter() {
  let count = 0;            // private state
  return {
    increment() { count++; },
    decrement() { count--; },
    value()     { return count; }
  };
}

const counter = makeCounter();
counter.increment();
counter.increment();
console.log(counter.value());  // 2
// count is private — not accessible directly

Practical uses: data privacy, memoization, factory functions, event listeners, partial application.

Q6 Classic: What does this loop print? How do you fix it?
Intermediate▾
Answer
// Problem — prints 3, 3, 3
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// All callbacks share same `i` (var is function-scoped)

// Fix 1: use let (block-scoped, new binding per iteration)
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);  // 0, 1, 2
}

// Fix 2: IIFE to capture each i
for (var i = 0; i < 3; i++) {
  ((function(j) {
    setTimeout(() => console.log(j), 100);
  })(i));
}
Q7 What is the scope chain? How does JavaScript resolve variable lookups?
Intermediate▾
Answer

When JavaScript resolves a variable, it starts in the current scope and walks up through parent scopes until it reaches the global scope. This chain of scopes is the scope chain. If the variable isn't found anywhere, a ReferenceError is thrown.

const global = "global";

function outer() {
  const outerVar = "outer";
  function inner() {
    const innerVar = "inner";
    console.log(innerVar);  // "inner"   ← own scope
    console.log(outerVar);  // "outer"   ← parent scope
    console.log(global);    // "global"  ← global scope
  }
  inner();
}
outer();

Scope chains are determined lexically (at author-time), not dynamically at runtime.

JavaScript — Async & Promises

Q8 Explain the JavaScript Event Loop. What is the call stack, task queue, and microtask queue?
Intermediate▾
Answer

JavaScript is single-threaded. The Call Stack executes synchronous code. When the stack is empty, the Event Loop checks queues:

  • Microtask Queue (higher priority): Promise callbacks (.then), queueMicrotask, MutationObserver
  • Task Queue (Macrotask): setTimeout, setInterval, I/O callbacks, UI events

Microtasks are fully drained before the next macrotask runs.

console.log("1");                             // sync
setTimeout(() => console.log("2"), 0);        // macrotask
Promise.resolve().then(() => console.log("3")); // microtask
console.log("4");                             // sync
// Output: 1, 4, 3, 2
Q9 Promises vs async/await — when would you use each?
Intermediate▾
Answer

async/await is syntactic sugar over Promises. It makes asynchronous code read like synchronous code and is preferred for sequential operations. Promise chains shine for parallel operations and complex flow control.

// Promise chain
fetchUser(id)
  .then(user => fetchPosts(user.id))
  .then(posts => render(posts))
  .catch(err => handleError(err));

// async/await — same logic, more readable
async function loadUserPosts(id) {
  try {
    const user  = await fetchUser(id);
    const posts = await fetchPosts(user.id);
    render(posts);
  } catch (err) {
    handleError(err);
  }
}

// Parallel with Promise.all
const [users, products] = await Promise.all([
  fetchUsers(), fetchProducts()
]);
Q10 Explain Promise.all, Promise.allSettled, Promise.race, and Promise.any.
Advanced▾
Answer
const p1 = Promise.resolve(1);
const p2 = Promise.reject("fail");
const p3 = Promise.resolve(3);

// all — rejects immediately if ANY rejects
Promise.all([p1, p2, p3]).catch(e => console.log(e)); // "fail"

// allSettled — waits for ALL, returns status+value for each
Promise.allSettled([p1, p2, p3]).then(results => {
  // [{status:"fulfilled",value:1},{status:"rejected",reason:"fail"},...]
});

// race — resolves/rejects with FIRST settled promise
Promise.race([p1, p2]).then(v => console.log(v)); // 1

// any — resolves with FIRST fulfilled; rejects if ALL reject
Promise.any([p2, p3]).then(v => console.log(v));  // 3

JavaScript — Prototypes & OOP

Q11 Explain prototypal inheritance. How does the prototype chain work?
Intermediate▾
Answer

Every JavaScript object has an internal [[Prototype]] link to another object. When accessing a property, JS first checks the object itself, then walks up the prototype chain until it finds it or reaches null.

function Animal(name) { this.name = name; }
Animal.prototype.speak = function() {
  return `${this.name} makes a sound`;
};

function Dog(name) { Animal.call(this, name); }
Dog.prototype = Object.create(Animal.prototype);
Dog.prototype.constructor = Dog;
Dog.prototype.speak = function() {
  return `${this.name} barks`;
};

const d = new Dog("Rex");
console.log(d.speak());         // "Rex barks"
console.log(d instanceof Animal); // true
Q12 Explain this in JavaScript. How does it behave in different contexts?
Intermediate▾
Answer

this is determined at call-time, not definition-time (except for arrow functions which lexically bind this).

// 1. Method call — this = owning object
const obj = { name:"Alice", greet() { return this.name; } };
obj.greet(); // "Alice"

// 2. Regular function call — this = undefined (strict) or global
function who() { return this; }
who(); // window / undefined

// 3. Arrow function — this = enclosing lexical context
const timer = {
  id: 42,
  start() {
    setTimeout(() => console.log(this.id), 100); // 42
  }
};

// 4. Explicit binding — call, apply, bind
function greet(greeting) { return `${greeting}, ${this.name}`; }
greet.call({name:"Bob"}, "Hi");  // "Hi, Bob"

Q13 Explain destructuring, spread (...), and rest parameters with practical examples.
Basic▾
Answer
// Destructuring — arrays
const [first, second, ...rest] = [1, 2, 3, 4];
// first=1, second=2, rest=[3,4]

// Destructuring — objects (with rename + default)

const newObj = { ...obj, extra: 5 }; // shallow clone+extend

}
sum(1, 2, 3, 4); // 10
Q14 Explain map, filter, and reduce. When would you use each?
Basic▾
Answer
const nums = [1, 2, 3, 4, 5];

// map — transform each element → new array (same length)
const doubled = nums.map(n => n * 2);  // [2,4,6,8,10]

const evens = nums.filter(n => n % 2 === 0);  // [2,4]

  .filter(n => n % 2 === 0)
  .map(n => n * 2)
  .reduce((acc, n) => acc + n, 0);  // 12

JavaScript — Advanced

Q15 Implement debounce and throttle from scratch. When would you use each?
Advanced▾
Answer

Debounce: executes after a delay from the last call — good for search-as-you-type, resize handlers. Throttle: executes at most once per interval — good for scroll events, mousemove.

// Debounce
function debounce(fn, delay) {
  let timer;
  return function(...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// Throttle
function throttle(fn, limit) {
  let lastCall = 0;
  return function(...args) {
    const now = Date.now();
    if (now - lastCall >= limit) {
      lastCall = now;
      return fn.apply(this, args);
    }
  };
}

// Usage
const onSearch = debounce(val => fetchResults(val), 300);
const onScroll = throttle(() => updatePosition(), 100);
Q16 What is memoization? Implement a generic memoize function.
Advanced▾
Answer

Memoization caches the results of expensive function calls so repeated calls with the same arguments return the cached result without recomputation.

function memoize(fn) {
  const cache = new Map();
  return function(...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

const fib = memoize(function(n) {
  return n <= 1 ? n : fib(n - 1) + fib(n - 2);
});

console.log(fib(40)); // Fast — results cached
Q17 What are generators? How do they differ from regular functions?
Advanced▾
Answer

Generator functions (function*) can be paused and resumed via yield. They return an iterator that produces values lazily — useful for infinite sequences and cooperative multitasking.

function* idGenerator() {
  let id = 1;
  while (true) yield id++;     // infinite, but lazy
}

const gen = idGenerator();
console.log(gen.next().value); // 1
console.log(gen.next().value); // 2

// Range generator
function* range(start, end, step = 1) {
  for (let i = start; i < end; i += step) yield i;
}
console.log([...range(0, 10, 2)]); // [0,2,4,6,8]
Q18 When would you use Map/Set over plain objects/arrays? What is WeakMap?
Advanced▾
Answer

Map: any value as key (including objects), preserves insertion order, O(1) lookups. Set: unique values only — great for deduplication. WeakMap/WeakSet: keys must be objects and are held weakly — won't prevent garbage collection, non-iterable. Ideal for caching metadata without memory leaks.

// Set deduplication
const unique = [...new Set([1, 2, 2, 3])];  // [1,2,3]

// Map with object keys
const cache = new Map();
const domEl = document.querySelector("#app");
cache.set(domEl, { clicks: 0 });

// WeakMap — metadata that doesn't prevent GC
const meta = new WeakMap();
function trackObject(obj) {
  meta.set(obj, { created: Date.now() });
}
// When obj is GC'd, the metadata entry is automatically removed
Q19 What is the difference between deep clone and shallow clone? How do you deep clone?
Intermediate▾
Answer
const obj = { a: 1, nested: { b: 2 } };

// Shallow clone — nested is still shared reference
const shallow = { ...obj };
shallow.nested.b = 99;
console.log(obj.nested.b); // 99 (mutated!)

// Deep clone options
// 1. structuredClone (modern, handles Date/Map/Set)
const deep1 = structuredClone(obj);

// 2. JSON round-trip (simple, but loses functions/undefined/Date)
const deep2 = JSON.parse(JSON.stringify(obj));

// 3. Recursive implementation
function deepClone(val) {
  if (val === null || typeof val !== "object") return val;
  if (Array.isArray(val)) return val.map(deepClone);
  return Object.fromEntries(
    Object.entries(val).map(([k, v]) => [k, deepClone(v)])
  );
}
Q20 What are Proxy and Reflect? Give a real-world use case.
Advanced▾
Answer

Proxy intercepts object operations (get, set, delete, etc.). Reflect provides default implementations for those operations. Used in reactive systems (Vue 3's reactivity), validation, logging.

// Validation proxy
function createValidator(target, rules) {
  return new Proxy(target, {
    set(obj, prop, value) {
      if (rules[prop] && !rules[prop](value)) {
        throw new TypeError(`Invalid value for ${prop}`);
      }
      return Reflect.set(obj, prop, value);
    }
  });
}

const user = createValidator({}, {
  age: v => typeof v === "number" && v >= 0
});
user.age = 25;   // ✓
// user.age = -1;  // TypeError

React — Fundamentals

R1 What is the Virtual DOM and how does React's reconciliation algorithm work?
Basic▾
Answer

The Virtual DOM is a lightweight in-memory representation of the real DOM. When state changes, React:

  1. Creates a new Virtual DOM tree
  2. Diffs it against the previous tree (reconciliation)
  3. Computes the minimal set of real DOM operations needed (the "diff")
  4. Applies those changes in a single batch (commit phase)

React's heuristic diff algorithm is O(n) — it assumes elements of different types produce different trees, and uses key props to match list items across renders.

Key insight: Stable key values on list items prevent unnecessary unmounting/remounting. Never use array index as key when the list can reorder.
R2 What is the difference between controlled and uncontrolled components?
Basic▾
Answer

In a controlled component, form data is handled by React state — the component is the "single source of truth". In an uncontrolled component, data is handled by the DOM itself, accessed via ref.

// Controlled — React owns the value
function ControlledInput() {
  const [value, setValue] = useState("");
  return <input value={value} onChange={e => setValue(e.target.value)} />;
}

// Uncontrolled — DOM owns the value
function UncontrolledInput() {
  const inputRef = useRef(null);
  const handleSubmit = () => console.log(inputRef.current.value);
  return <input ref={inputRef} defaultValue="hello" />;
}

Prefer controlled components — they're easier to validate and test. Use uncontrolled for file inputs or when integrating with non-React libraries.

R3 What is the difference between props and state? When should you lift state up?
Basic▾
Answer

Props are inputs passed from parent to child — read-only for the child. State is data owned and managed by a component — mutable via setState/setter.

Lift state up when multiple sibling components need to share/react to the same data — move the state to their closest common ancestor and pass it down via props.

// Before: each component has its own count
// After: parent owns the shared count
function Parent() {
  const [count, setCount] = useState(0);
  return (
    <>
      <Display count={count} />
      <Controls onIncrement={() => setCount(c => c + 1)} />
    </>
  );
}

React — Hooks

R4 Explain useEffect — dependencies, cleanup, and common pitfalls.
Intermediate▾
Answer
// Runs after every render
useEffect(() => { ... });

// Runs once on mount (empty dep array)
useEffect(() => { ... }, []);

// Runs when `userId` changes
useEffect(() => {
  let cancelled = false;
  fetchUser(userId).then(data => {
    if (!cancelled) setUser(data);
  });
  return () => { cancelled = true; }; // cleanup — cancel stale request
}, [userId]);

// Event listener with cleanup
useEffect(() => {
  const handler = (e) => setKey(e.key);
  window.addEventListener("keydown", handler);
  return () => window.removeEventListener("keydown", handler);
}, []);
Common pitfalls: Missing dependencies (stale closures), objects/arrays in deps triggering infinite loops, forgetting cleanup for subscriptions/timers.
R5 When should you use useCallback vs useMemo?
Intermediate▾
Answer

useMemo memoizes a computed value. useCallback memoizes a function reference. Both only recompute/recreate when dependencies change. Use them when the cost of recreation is high or when passing to memoized children.

function ProductList({ products, minPrice }) {
  // useMemo — expensive filtering
  const filtered = useMemo(
    () => products.filter(p => p.price >= minPrice),
    [products, minPrice]
  );

  // useCallback — stable ref for child component
  const handleBuy = useCallback((productId) => {
    addToCart(productId);
  }, []);  // recreate only if addToCart changes

  return filtered.map(p =>
    <ProductCard key={p.id} product={p} onBuy={handleBuy} />
  );
}
Don't over-optimize. useMemo/useCallback add overhead. Profile first with React DevTools before adding them everywhere.
R6 What are the different uses of useRef beyond DOM access?
Intermediate▾
Answer
// 1. DOM access
const inputRef = useRef(null);
<input ref={inputRef} />
inputRef.current.focus();

// 2. Persist mutable value across renders (no re-render on change)
const renderCount = useRef(0);
useEffect(() => { renderCount.current++; });

// 3. Store previous value
function usePrevious(value) {
  const ref = useRef();
  useEffect(() => { ref.current = value; }, [value]);
  return ref.current;
}

// 4. Store timer/subscription IDs
const timerRef = useRef(null);
timerRef.current = setInterval(...);
return () => clearInterval(timerRef.current);
R7 What are custom hooks? Build a useFetch hook from scratch.
Intermediate▾
Answer

Custom hooks are functions starting with use that extract and share stateful logic between components without changing component hierarchy.

function useFetch(url) {
  const [state, setState] = useState({
    data: null, loading: true, error: null
  });

  useEffect(() => {
    let cancelled = false;
    setState({ data: null, loading: true, error: null });

    fetch(url)
      .then(r => r.json())
      .then(data => { if (!cancelled) setState({ data, loading: false, error: null }); })
      .catch(err  => { if (!cancelled) setState({ data: null, loading: false, error: err }); });

    return () => { cancelled = true; };
  }, [url]);

  return state;
}

// Usage
const { data, loading, error } = useFetch("/api/users");

React — State Management

R8 When would you use useReducer instead of useState?
Intermediate▾
Answer

Use useReducer when: state is an object with multiple sub-values, next state depends on current state in complex ways, or you have many related state transitions.

const initialState = { count: 0, step: 1, history: [] };

function reducer(state, action) {
  switch (action.type) {
    case "INCREMENT":
      return { ...state, count: state.count + state.step,
               history: [...state.history, state.count] };
    case "SET_STEP":
      return { ...state, step: action.payload };
    case "RESET":
      return initialState;
    default:
      return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, initialState);
  return (
    <button onClick={() => dispatch({ type: "INCREMENT" })}>
      Count: {state.count}
    </button>
  );
}
R9 Context API vs Redux — when would you choose each?
Intermediate▾
Answer

Context API is built into React, zero setup, great for low-frequency updates (theme, auth, locale). Its downside: all consumers re-render when context value changes — can cause performance issues with frequently-changing state.

Redux (or Zustand/Jotai) shines when: you have complex state logic, need time-travel debugging, have high-frequency updates, or need fine-grained subscriptions to prevent unnecessary re-renders.

// Context — simple theme toggle
const ThemeCtx = createContext("light");

function App() {
  const [theme, setTheme] = useState("light");
  return (
    <ThemeCtx.Provider value={{ theme, setTheme }}>
      <Children />
    </ThemeCtx.Provider>
  );
}

function Button() {
  const { theme, setTheme } = useContext(ThemeCtx);
  return <button onClick={() => setTheme(t => t === "light" ? "dark" : "light")}>Toggle</button>;
}

React — Performance

R10 What is React.memo? When does it help and when does it not?
Intermediate▾
Answer

React.memo is a HOC that prevents re-rendering if props haven't changed (shallow comparison). It helps when a child is expensive to render and its parent re-renders frequently.

// Without memo — re-renders every time Parent renders
const ExpensiveChild = ({ data }) => <HeavyChart data={data} />;

// With memo — only re-renders when `data` changes
const ExpensiveChild = React.memo(({ data }) => <HeavyChart data={data} />);

// Custom comparison (for deep equality)
const MemoComp = React.memo(Comp, (prevProps, nextProps) => {
  return prevProps.items.length === nextProps.items.length;
});
Gotcha: React.memo does a shallow comparison. Objects/arrays created inline (e.g. style={{color:'red'}}) are always new references — wrap the parent's callbacks in useCallback and objects in useMemo.

React — Patterns

R12 Explain the Compound Component pattern with an example.
Advanced▾
Answer

Compound components share implicit state via context, giving consumers flexible composition without exposing internal details (think <select>/<option>).

const TabsCtx = createContext();

function Tabs({ children, defaultTab }) {
  const [active, setActive] = useState(defaultTab);
  return (
    <TabsCtx.Provider value={{ active, setActive }}>
      {children}
    </TabsCtx.Provider>
  );
}

Tabs.Tab = ({ id, children }) => {
  const { active, setActive } = useContext(TabsCtx);
  return <button onClick={() => setActive(id)} className={active === id ? "active" : ""}>{children}</button>;
};

Tabs.Panel = ({ id, children }) => {
  const { active } = useContext(TabsCtx);
  return active === id ? <div>{children}</div> : null;
};

// Usage
<Tabs defaultTab="a">
  <Tabs.Tab id="a">Tab A</Tabs.Tab>
  <Tabs.Panel id="a">Content A</Tabs.Panel>
</Tabs>
R13 What is the render props pattern? How has it been superseded by hooks?
Intermediate▾
Answer

Render props is a technique where a component receives a function as a prop and calls it with its state, allowing the consumer to control rendering.

// Render prop version
function MouseTracker({ render }) {
  const [pos, setPos] = useState({ x: 0, y: 0 });
  return <div onMouseMove={e => setPos({ x: e.clientX, y: e.clientY })}>
    {render(pos)}
  </div>;
}
// Usage: <MouseTracker render={pos => <p>{pos.x},{pos.y}</p>} />

// Modern hook version — same logic, simpler API
function useMouse() {
  const [pos, setPos] = useState({ x: 0, y: 0 });
  useEffect(() => {
    const h = e => setPos({ x: e.clientX, y: e.clientY });
    window.addEventListener("mousemove", h);
    return () => window.removeEventListener("mousemove", h);
  }, []);
  return pos;
}

React — Advanced

R14 What is React Concurrent Mode? Explain useTransition and useDeferredValue.
Advanced▾
Answer

Concurrent Mode lets React work on multiple versions of the UI simultaneously and interrupt, pause, or abandon renders. It enables prioritizing urgent updates (typing) over non-urgent ones (search results).

// useTransition — mark update as non-urgent
function SearchPage() {
  const [query, setQuery] = useState("");
  const [results, setResults] = useState([]);
  const [isPending, startTransition] = useTransition();

  const handleChange = (e) => {
    setQuery(e.target.value);        // urgent — instant
    startTransition(() => {
      setResults(search(e.target.value)); // deferred — can be interrupted
    });
  };
  return <input value={query} onChange={handleChange} />;
}

// useDeferredValue — defer a value derived from props
function List({ query }) {
  const deferred = useDeferredValue(query);
  // deferred lags behind query; shows stale content while new content loads
  return <Results query={deferred} />;
}
R15 What are Error Boundaries? Implement one.
Intermediate▾
Answer

Error boundaries catch JavaScript errors anywhere in their child component tree, log them, and display a fallback UI. They must be class components (as of React 18).

class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, info) {
    logErrorToService(error, info.componentStack);
  }

  render() {
    if (this.state.hasError) {
      return <h2>Something went wrong. <button onClick={() => this.setState({ hasError: false })}>Retry</button></h2>;
    }
    return this.props.children;
  }
}

// Usage
<ErrorBoundary><MyWidget /></ErrorBoundary>
Note: Error boundaries do NOT catch async errors, event handler errors, or server-side rendering errors — use try/catch for those.

Node.js — Core Concepts

N1 How does Node.js achieve non-blocking I/O despite being single-threaded?
Basic▾
Answer

Node.js uses the libuv library which provides an event loop and a thread pool (default 4 threads). For I/O operations (file, network, DNS), Node offloads work to the OS or thread pool and registers a callback. The main thread is free to continue processing. When the I/O completes, the callback is pushed to the event queue.

This model excels at I/O-bound workloads but is unsuitable for CPU-intensive tasks without worker threads, since heavy computation blocks the event loop.

// Non-blocking — callback fires when file is read
const fs = require("fs");

fs.readFile("big.txt", "utf8", (err, data) => {
  if (err) throw err;
  console.log(data.slice(0, 100));
});

console.log("This runs BEFORE the file is done reading");
N2 What is the difference between CommonJS require and ES Modules import?
Basic▾
Answer
// CommonJS (CJS) — synchronous, dynamic, Node default
const fs = require("fs");
const utils = require("./utils");
module.exports = { myFn };

// ES Modules (ESM) — static, async, tree-shakeable
import fs from "node:fs";
import { helper } from "./utils.js";
export { myFn };

// Dynamic import (works in both ESM and script contexts)
const module = await import("./heavy.js");

Key differences: CJS loads synchronously and can be conditional. ESM is parsed statically at compile time, enabling tree-shaking and top-level await. Use ESM for new projects ("type":"module" in package.json).

N3 Explain process.nextTick vs setImmediate vs setTimeout(fn, 0).
Advanced▾
Answer
setImmediate(() => console.log("setImmediate"));
setTimeout(() => console.log("setTimeout 0"), 0);
process.nextTick(() => console.log("nextTick"));
Promise.resolve().then(() => console.log("Promise"));

// Output: nextTick → Promise → setTimeout/setImmediate*
// (* setTimeout vs setImmediate order can vary)
  • process.nextTick: fires before any I/O event, before the event loop continues. Use for deferring callbacks after the current operation completes.
  • Promise microtasks: drain after nextTick queue, before I/O.
  • setImmediate: fires in the check phase of the event loop, after I/O callbacks.
  • setTimeout(fn, 0): fires in the timers phase, minimum ~1ms delay.

N5 When would you use Worker Threads in Node.js?
Advanced▾
Answer

Worker Threads enable CPU-bound parallelism without blocking the event loop. Use them for: image/video processing, crypto operations, parsing large datasets, ML inference.

// worker.js
const { workerData, parentPort } = require("worker_threads");
const result = expensiveComputation(workerData);
parentPort.postMessage(result);

// main.js
const { Worker } = require("worker_threads");

function runInWorker(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker("./worker.js", { workerData: data });
    worker.on("message", resolve);
    worker.on("error",   reject);
    worker.on("exit",    code => {
      if (code !== 0) reject(new Error(`Worker exited: ${code}`));
    });
  });
}

Middleware functions have signature (req, res, next). They can modify req/res, end the request-response cycle, or call next() to pass control. Error-handling middleware has 4 arguments: (err, req, res, next).

    error: err.message || "Internal Server Error",
    ...(process.env.NODE_ENV === "development" && { stack: err.stack })
  });
};

N9 How do you scale a Node.js application? Explain clustering.
Advanced▾
Answer

Node.js runs on a single core by default. The cluster module spawns worker processes that share a server port, using all CPU cores. In production, tools like PM2 handle clustering automatically.

const cluster = require("cluster");
const os = require("os");

if (cluster.isPrimary) {
  const numCPUs = os.cpus().length;
  console.log(`Master: forking ${numCPUs} workers`);

  for (let i = 0; i < numCPUs; i++) cluster.fork();

  cluster.on("exit", (worker, code) => {
    console.log(`Worker ${worker.process.pid} died. Restarting…`);
    cluster.fork();  // auto-restart
  });
} else {
  // Each worker runs the actual server
  require("./server");
}

// PM2 alternative (simpler)
// pm2 start server.js -i max
N10 What causes memory leaks in Node.js and how do you detect and fix them?

Answer

Common causes:

  • Global variables accumulating data
  • Event listeners not removed (emitter.removeListener)
  • Closures holding large objects unnecessarily
  • Uncleaned timers/intervals

N13 What is connection pooling and why is it important in Node.js?
Intermediate▾
Answer

Creating a new DB connection per request is expensive (~50–100ms). A connection pool maintains a set of reusable connections. Requests borrow a connection from the pool, use it, and return it — dramatically reducing latency.

// pg (PostgreSQL)
const { Pool } = require("pg");
const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  max: 20,              // max connections
  idleTimeoutMillis: 30_000,
  connectionTimeoutMillis: 2_000
});

// Reuse the pool across your app (singleton)
async function query(text, params) {
  const client = await pool.connect();
  try {
    return await client.query(text, params);
  } finally {
    client.release(); // ALWAYS release back to pool
  }
}
